## Deletion strategy

- **Default**: Prefer **soft delete** over physical (`DELETE` / `session.delete`) removal for domain entities that represent business records, accounts, orders, content, or anything that may need audit, restore, or referential integrity over time.
- **Meaning**: A row stays in the table but is marked as no longer active for normal reads and writes. List/detail APIs and repository queries should treat soft-deleted rows as **absent** unless the operation is explicitly administrative or forensic.

## Schema convention

- Use a **nullable timestamp** `deleted_at` (timezone-aware) on the model. `NULL` means “not deleted”; a non-`NULL` value means “deleted at this instant.”
- Prefer `deleted_at` over a bare `is_deleted` flag unless you have a strong reason—timestamps support auditing and delayed purge jobs.
- Add `deleted_at` to indexes only when query patterns require it (e.g. filtering “active” rows often benefits from a composite index with tenant/status fields).

## Queries and repositories

- **Default filter**: Select only rows where `deleted_at IS NULL` for normal business operations. Centralize this in `repository.py` (e.g. shared query helpers or base filters) so routers and services do not repeat the condition everywhere.
- **Delete operation**: Implement “delete” as `UPDATE ... SET deleted_at = now()` (or service-layer equivalent), not `session.delete()`, unless the row is excluded by the exceptions below.
- **Restore**: If product requirements include undelete, expose it explicitly in service/API and clear `deleted_at` in one place with appropriate authorization checks.

## Uniqueness and constraints

- Natural or business keys (e.g. email) that must be unique among **active** rows only should use a **partial unique index** in PostgreSQL (e.g. unique where `deleted_at IS NULL`), not a naive global `UNIQUE` on the column if reuse of values after soft delete is allowed.
- If the business forbids reusing a key even after soft delete, a global unique constraint can remain; document that choice.

## Relationships

- Prefer soft delete on the **owned** entity; avoid relying on `ON DELETE CASCADE` for primary business removal flows.
- Foreign keys to soft-deleted parents: define expected behavior (e.g. child rows remain but are unreachable through normal queries, or children are soft-deleted in the same transaction when required by domain rules).

## When hard delete is acceptable

- **Join / association tables** with no standalone meaning, if orphan rows are useless and retention is not required.
- **Idempotent cleanup** of transient data (sessions, jobs, cache-like tables) where no audit trail is needed.
- **Legal or compliance**: explicit erasure (“right to be forgotten”) may require hard delete or anonymization; treat that as a dedicated purge flow, not the default `DELETE` endpoint behavior.

## API and errors

- Default **user-facing delete** should behave like removal: subsequent GET/list as a normal client should not return the row (often **404** for GET-by-id, omit from list results).
- Administrative or support endpoints that must see soft-deleted rows should be clearly separated (route prefix, permission, or flag) and documented.

## Alembic migrations

- **Do not hand-author new revision files** (no new empty migration created by copy-paste or from scratch). Always **generate** revisions with autogenerate after ORM/model changes.
- **Command**: `make revision m="short description"` — runs `alembic revision --autogenerate -m "..."` (see `Makefile`).
- **Workflow**: update SQLAlchemy models first, ensure models are imported in `alembic/env.py`, then run `make revision m="..."`, review the generated file, apply with `make migrate`.
- **Edits inside a generated file** are allowed only for what autogenerate cannot do (data backfills, partial indexes, careful `op.execute`, etc.). Prefer extending the **autogenerated** revision rather than adding a separate hand-written migration file for the same schema change.

## Migrations and tests

- Adding `deleted_at` requires an Alembic migration; backfill existing rows to `NULL`.
- Tests that assert deletion should verify behavior through the **public API** and repository filters (row still exists in DB when soft-deleted, invisible to normal queries), unless the scenario explicitly targets physical removal.
