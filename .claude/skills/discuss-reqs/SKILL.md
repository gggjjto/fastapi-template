# Discuss Requirements

需求来自他人或尚不完整时使用。目标是在写代码前，通过来回讨论把所有盲点、边界情况、隐含假设都浮出水面。

---

## 我的角色

你描述需求，我来分析：

- 当前描述里**已经清楚**的是什么
- **不清楚或缺失**的是什么
- 你**可能没有想到**的维度

对于有标准答案的问题（如软删除、分页、认证），我直接给出推荐并说明理由，不留开放性问题。对于有真实业务权衡的问题，我们一起讨论。

---

## 讨论维度（我会主动检查每一项）

### 1. 功能边界
- 需要哪些操作？只有增删改查的一部分吗？
- 需要批量操作、导出、搜索、过滤、排序吗？
- 有没有"我以为不需要但其实需要"的操作？

### 2. 认证与权限
- 谁可以执行这个操作？已登录用户？特定角色？
- 用户能操作自己的资源还是所有人的？（所有权检查）
- 有没有管理员和普通用户不同权限的情况？

### 3. 数据模型
- 涉及哪些字段？类型？校验规则（长度、格式、范围）？
- 哪些字段是必填，哪些是可选？
- 有没有联合唯一约束？是全局唯一还是仅在活跃记录中唯一？
- 删除策略：软删除（`deleted_at`）还是物理删除？（默认软删除）

### 4. 关联关系
- 这个资源和其他域有关联吗？
- 被关联资源删除时，这里怎么处理？（级联？软删除？报错？）
- 查询时需要返回关联数据吗？（注意 N+1 风险）

### 5. 错误场景
- 资源不存在时返回什么？（404）
- 冲突时怎么处理？（409）
- 无权限时应该 401 还是 403？
- 有没有业务规则导致的自定义错误？

### 6. 并发与幂等
- 同一个操作两次提交会怎样？需要幂等吗？
- 有没有竞态条件？（比如两个用户同时操作同一资源）

### 7. 列表与分页
- 列表端点需要分页吗？（超过几十条就应该分页）
- 需要哪些过滤、排序参数？
- 默认排序是什么？

### 8. 迁移与兼容性
- 需要数据库 migration 吗？
- 有没有存量数据需要处理？
- 这个改动是否影响现有 API 合约？

### 9. 性能
- 预期数据量大吗？需要加索引吗？
- 有没有可以缓存的查询？

---

## 讨论节奏

每轮最多提 **3-5 个问题**，不要一次性抛出所有问题。优先问影响最大的盲点。

每轮结束后，等你回答，再进入下一轮。

达到以下条件才结束讨论：

- 所有端点的输入、输出、错误路径都明确
- 认证和权限规则清楚
- 数据模型字段和约束确定
- 是否需要 migration 确定
- 没有"可能"或"大概"的模糊描述

---

## 最终输出

讨论完成后，输出确认的规格块：

```
Domain: <name>
Endpoints:
  - <METHOD /path — 一句话说明>
Auth: <public | CurrentUser | 角色限制>
Fields:
  - <field: type, 约束>
Errors:
  - <场景 → HTTP code>
Response: <data 字段结构>
Migration: <yes | no，如果 yes 说明变更内容>
Edge cases resolved:
  - <已确认的边界情况>
```

问："这个规格你确认了吗？可以开始写代码了吗？"

确认后，将需求追加到 `.claude/requirements.md`，然后运行 `/feature`。
