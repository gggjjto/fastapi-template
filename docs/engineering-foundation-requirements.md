# Engineering Foundation Requirements

## Related Documents

- `docs/fastapi-saas-infrastructure-architecture-blueprint.md` — high-level SaaS infrastructure and application architecture blueprint.
- `docs/engineering-foundation-design.md` — technical design and implementation plan for these requirements.

## 1. Background

The project is a domain-oriented FastAPI backend template with existing support for unified responses, JWT authentication, SQLAlchemy, Alembic, Redis, Arq, structlog, Sentry, rate limiting, integration tests, Docker, and GitHub Actions.

The next goal is to evolve the template into a production-ready foundation for long-lived SaaS and large internet-style backend projects. This work focuses on infrastructure and architecture capabilities rather than business features.

## 2. Goals

- Provide stable and consistent API contracts for frontend and external clients.
- Centralize cross-cutting concerns such as errors, logging, configuration, authentication, authorization, documentation, and internationalization.
- Improve production readiness, security, observability, and maintainability.
- Keep the application as a modular monolith until service boundaries, team ownership, or scale pressure justify extraction.
- Produce changes that can be implemented incrementally by AI agents without broad rewrites.

## 3. Non-Goals

- Do not migrate from SQLAlchemy to another ORM.
- Do not use Aerich. This project uses SQLAlchemy, so database migrations must use Alembic.
- Do not introduce microservices at this stage.
- Do not introduce a heavy authorization framework unless the lightweight RBAC model proves insufficient.
- Do not implement billing, subscription, or full multi-tenant SaaS business flows in this phase.
- Do not expose production OpenAPI documentation publicly by default.

## 4. Scope

This initiative covers:

- Unified API response envelope.
- Business error code system.
- Global exception handling.
- Production logging and request context.
- Environment-aware configuration management.
- Alembic migration baseline and migration rules.
- JWT authentication hardening.
- RBAC authorization model.
- Internationalized error messages.
- OpenAPI documentation standards.

## 5. Current State Summary

- `ApiResponse[T]` exists and returns `{code, message, data}`.
- Global handlers exist for `HTTPException` and `RequestValidationError`.
- Domain exceptions exist but are not globally handled.
- Logging uses `structlog` and request ID middleware.
- Settings use `pydantic-settings` with `APP_` prefix and `APP_ENV`.
- Alembic is configured, but there is no initial migration file.
- JWT access and refresh tokens exist.
- `/auth/me` requires authentication, but user APIs are still broadly open.
- RBAC is not implemented.
- i18n is not implemented.
- OpenAPI is generated by FastAPI but lacks a project-wide documentation standard.

## 6. Requirements

### REQ-001: Unified Response Envelope

Priority: P0

The API must return a consistent response envelope for successful responses and known error responses.

Requirements:

- Keep a single reusable response model in `app/core/response.py`.
- Successful responses must be generated via `ApiResponse.ok(...)`.
- Error responses must be generated by exception handlers, not manually in routers.
- The response envelope must support a stable business code, message, data, and request ID.
- The envelope must remain compatible with FastAPI response models and OpenAPI generation.

Acceptance Criteria:

- All existing endpoints still return a consistent envelope.
- Known errors include a request ID when available.
- Tests assert at least one success response and one error response shape.

### REQ-002: Business Error Code System

Priority: P0

The project must define stable business error codes that are independent from translated messages.

Requirements:

- Define common error codes in `app/core/error_codes.py`.
- Keep domain-specific error codes near each domain, such as `app/auth/constants.py` and `app/users/constants.py`.
- Error codes must use stable uppercase string identifiers, such as `AUTH_INVALID_TOKEN` or `USER_NOT_FOUND`.
- HTTP status codes must continue to represent protocol-level status.
- Business code must represent application-level failure reason.

Acceptance Criteria:

- Core, auth, and users errors have stable error code constants.
- Existing domain exceptions expose error codes.
- Error code values are tested and documented.

### REQ-003: Global Exception Handling

Priority: P0

The project must centralize known exception handling.

Requirements:

- `DomainError` must carry at least `code`, `message`, and `status_code`.
- Register a global `DomainError` exception handler.
- Existing routers should not manually translate domain exceptions unless there is a route-specific reason.
- Preserve handlers for `HTTPException`, `RequestValidationError`, `RateLimitExceeded`, and unexpected exceptions.
- Unexpected exceptions must be logged with request context and return a generic response.

Acceptance Criteria:

- Domain exceptions are converted by the global handler.
- Validation errors keep a predictable shape.
- Unknown exceptions do not leak stack traces to clients.
- Tests cover domain, validation, HTTP, rate limit, and unexpected error handling where practical.

### REQ-004: Logging and Request Context

Priority: P0

The project must provide production-ready structured logging.

Requirements:

- Continue using `structlog`.
- Every request must have a request ID.
- Request logs must include method, path, status code, duration, and request ID.
- When authenticated context is available, logs should support user ID and tenant ID fields.
- Error logs must include error code and exception details.
- Sensitive values such as passwords, tokens, cookies, and secrets must never be logged.

Acceptance Criteria:

- Request middleware emits one access log per request.
- Error handler logs include request ID and error code.
- Tests verify `X-Request-ID` propagation.
- Production configuration can force JSON logs.

### REQ-005: Environment-Aware Configuration

Priority: P0

Configuration must be safe by default across development, test, staging, and production.

Requirements:

- Keep `pydantic-settings` and `APP_` environment variable prefix.
- Support `development`, `test`, `staging`, and `production`.
- Local `.env` usage is allowed only for local development.
- Production and staging secrets must be provided by environment or a secret manager.
- Production startup must fail fast when unsafe defaults are detected.

Production Safety Rules:

- `APP_JWT_SECRET` must not use the default value.
- `APP_ALLOWED_ORIGINS` must not be `["*"]`.
- `APP_DB_CREATE_TABLES_ON_STARTUP` must be `false`.
- `APP_DATABASE_URL` must use PostgreSQL.
- JSON logging should be enabled.
- OpenAPI must be disabled or protected.

Acceptance Criteria:

- Invalid production settings raise a clear configuration error.
- Tests cover at least default secret rejection and wildcard CORS rejection in production.
- README or docs describe environment expectations.

### REQ-006: Alembic Migration Baseline

Priority: P0

Database schema changes must be managed by Alembic.

Requirements:

- Add an initial Alembic migration for the current schema.
- Keep `APP_DB_CREATE_TABLES_ON_STARTUP=false` for production-like environments.
- All future model changes must include migration files.
- CI must continue verifying `alembic upgrade head` and downgrade where supported.
- Large or risky migrations must follow expand-contract rollout.

Acceptance Criteria:

- `alembic/versions/` contains an initial migration.
- Tests and CI can apply migrations from an empty database.
- Production docs no longer rely on `create_all`.

### REQ-007: JWT Authentication Hardening

Priority: P0

JWT authentication must support production session control.

Requirements:

- Keep short-lived access tokens.
- Add server-side refresh token session tracking.
- Store only refresh token hashes, never raw refresh tokens.
- Support refresh token rotation.
- Support logout by revoking the active session.
- Support revoking all sessions for a user.
- Detect refresh token reuse when possible.

Acceptance Criteria:

- Login creates a session record.
- Refresh rotates the refresh token.
- Logout revokes the session.
- Reusing a revoked refresh token fails.
- Tests cover login, refresh, rotation, logout, and revoked token rejection.

### REQ-008: RBAC Authorization

Priority: P0

The project must support lightweight role-based authorization.

Requirements:

- Add models for roles, permissions, user-role assignments, and role-permission assignments.
- Define permission strings using `resource:action`, such as `users:read`.
- Add reusable dependencies such as `RequirePermission("users:read")`.
- Router handlers declare required permissions.
- Service layer may still enforce resource-level ownership checks.
- Initial implementation can be global RBAC. Tenant-scoped RBAC can be added later.

Acceptance Criteria:

- Protected user list/detail endpoints require a permission.
- Unauthorized users receive 401.
- Authenticated users without permission receive 403.
- Tests cover allowed and denied access.

### REQ-009: Internationalized Error Messages

Priority: P1

Known error messages must support localization.

Requirements:

- Add a small i18n module under `app/core/i18n.py`.
- Support at least `zh-CN` and `en-US`.
- Resolve locale from user preference when available, then `Accept-Language`, then default locale.
- Error codes must not be translated.
- Logs should prefer error codes and stable metadata over translated messages.

Acceptance Criteria:

- Known domain errors can return Chinese or English messages.
- Missing translations fall back to the default locale.
- Tests cover locale negotiation and fallback.

### REQ-010: OpenAPI Documentation Standard

Priority: P1

The project must define consistent API documentation rules.

Requirements:

- Every route must define `summary`, `description`, `response_model`, and tags.
- Common error responses must be reusable.
- Authentication requirements must appear in OpenAPI.
- Request and response examples should be provided for public or important endpoints.
- Pagination responses must be documented consistently.
- Production OpenAPI must remain disabled or protected.

Acceptance Criteria:

- Existing auth, users, and health endpoints follow the documentation standard.
- OpenAPI includes bearer authentication metadata.
- Tests or a snapshot check verify key schema properties.

## 7. Implementation Phases

### Phase 1: Contract and Error Foundation

Deliver:

- Business error code system.
- Enhanced `DomainError`.
- Global `DomainError` handler.
- Response envelope request ID support.
- Tests for error responses.

### Phase 2: Migration and Configuration Safety

Deliver:

- Initial Alembic migration.
- Production configuration validation.
- Documentation for environment rules.
- CI-compatible migration flow.

### Phase 3: Authentication and RBAC

Deliver:

- Refresh token session model.
- Token rotation and logout.
- RBAC models and permission dependencies.
- User endpoint authorization.
- Tests for auth and authorization.

### Phase 4: i18n and OpenAPI Standardization

Deliver:

- Locale negotiation and message catalog.
- Localized domain error messages.
- Route documentation standard.
- OpenAPI authentication and error response schemas.

### Phase 5: Logging and Observability Hardening

Deliver:

- Access log middleware fields.
- Error log metadata.
- Sensitive data redaction.
- Preparation hooks for future OpenTelemetry integration.

## 8. Testing Requirements

- Prefer integration tests using the existing PostgreSQL and Redis test stack.
- Use dependency overrides where needed.
- Do not mock database behavior for repository or service flows unless unavoidable.
- Cover success and failure paths for each cross-cutting capability.
- Maintain mypy and Ruff compliance.
- Run `make ci` before final handoff when practical.

## 9. Documentation Requirements

- Update README if setup, environment variables, auth behavior, migration flow, or API contract changes.
- Add examples for common error response and RBAC usage.
- Keep implementation notes close to the relevant docs.

## 10. AI Execution Notes

When an AI agent implements this initiative:

- Work phase by phase.
- Do not combine RBAC, i18n, migration, and logging changes in a single large PR.
- Preserve current domain-oriented structure.
- Prefer small, reviewable commits.
- Add or update tests in the same PR as each behavior change.
- Do not introduce Aerich.
- Do not replace Alembic, SQLAlchemy, FastAPI, or the existing response model unless explicitly requested.
